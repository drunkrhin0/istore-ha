# ADR-002: v2.1 Security Hardening and Code Quality

## Status
Accepted

## Date
2026-07-07

## Context

ADR-001 documented the v2.0 architecture. After a security audit and code review (July 2026), several hardening and simplification changes were made before shipping v2.1. This ADR records the design decisions behind those changes.

## Decisions

### D-1: Typed exceptions replace generic Exception in auth chain

**Problem:** The auth chain functions (`_login`, `_get_app_id`, etc.) raised generic `Exception` with string messages. The config flow caught `Exception` and classified errors as auth vs connection by checking if the message contained keywords like "login", "auth", "password" — fragile to API wording changes.

**Decision:** Define two exception types:

- `IStoreAuthError` — authentication failure (invalid credentials). Raised by `_login` only.
- `IStoreApiError` — API error or connection issue. Raised by all other auth helpers and API methods.

**Consequences:**
- Config flow catches `IStoreAuthError` → `invalid_auth`, `IStoreApiError` → `cannot_connect`
- No string-matching dependency on iStore's error wording
- All `Exception` raises in the API class methods also switched to `IStoreApiError` for consistency

### D-2: Error response bodies never logged

**Problem:** Six auth helpers fell back to the full API response body in error messages: `f"login failed: {body}"` or `f"...: {body.get('message', body)}"`. If the iStore API ever returned PII or structured data in error responses, it would appear in HA logs.

**Decision:** All auth error messages now use `body.get('message', f"code={body.get('code')}")` — prefer the human-readable `message` key, fall back to the numeric code, never include the full body dict.

**Endpoints fixed:** `_get_public_key`, `_login`, `_set_session`, `_get_app_id`, `_get_site_id`, `_get_device_id`.

### D-3: Debug log identifiers masked

**Problem:** `authenticate()` logged `appId`, `siteId`, and `mdmId` at DEBUG level in full. These are internal iStore infrastructure identifiers — if debug logs were shared publicly (GitHub issues, forums), they could be harvested for enumeration.

**Decision:** `_mask_id(value)` truncates identifiers: values ≤7 chars return `****`, longer values return first 4 + `...` + last 4 chars. Applied to all three identifiers in `authenticate()`.

### D-4: Shared aiohttp session for all API methods

**Problem:** Five of six API methods (`get_architecture`, `get_attributes`, `set_onoff`, `set_timers_batch`, `update_asset_name`) created a new `aiohttp.ClientSession` per call. Only `get_measurements` (the 30s poll) used the shared session from `_get_session()`. Each new session opened a new TCP+TLS connection.

**Decision:** All six methods now use `await self._get_session()`. One session, one connection pool. The 30-second total timeout (`_DEFAULT_TIMEOUT`) applies uniformly.

### D-5: `async_write_timer_settings` accepts coordinator explicitly

**Problem:** `async_write_timer_settings()` reached into `self.hass.data[DOMAIN]` to find its own coordinator — a reverse dependency that couples the API class to the integration's stored data structure.

**Decision:** The coordinator is now passed as an explicit parameter: `async_write_timer_settings(self, coordinator, updates)`. All callers (`switch.py`, `select.py`, `time.py`) pass `self.coordinator`. The API class no longer imports `DOMAIN`.

### D-6: `_handle_401` helper deduplicates re-auth boilerplate

**Problem:** Six API methods contained identical 401 handling: log warning → `await self.re_authenticate()` → update `headers["Authorization"]`. Three lines duplicated six times.

**Decision:** Extract `_handle_401(headers)` as a private method. It calls `re_authenticate()` (which remains `_auth_lock`-guarded) and mutates the `headers` dict in place. All six call sites reduced from 3 lines to 1.

**Log level:** Uses `DEBUG` rather than `INFO` — token expiry is expected behavior, not a notable event. This is consistent with the principle that re-auth is an implementation detail, not an operational concern.

### D-7: `_get_point_value` / `_get_top_bottom` helpers deduplicate coordinator access

**Problem:** Eight entity classes across five files contained the same `data[self.api.mdm_id]["points"][point]["value"]` access pattern (7 lines each) with identical `None`-on-missing-data behavior.

**Decision:** Two module-level helpers in `sensor.py`:

- `_get_point_value(coordinator, mdm_id, point)` — returns value or `None`. Used by IStoreSensor, IStoreStatusSensor, IStoreBinarySensor, BaseIStoreSwitch, IStoreWorkModeSelect, IStoreTimeEntity.
- `_get_top_bottom(coordinator, mdm_id)` — returns `(top, bottom)` as floats or `(None, None)`. Used by IStoreRemainingHotWater, IStoreRawHotVolume, IStoreShowerTimeRemaining.

Both catch `(KeyError, TypeError)` (and `ValueError` for float conversion) rather than bare `except Exception` — narrow enough to surface structural bugs while handling expected API data shape variations.

### D-8: pip-audit in CI pipeline

**Problem:** The CI pipeline (`ci.yml`) had no dependency vulnerability scanning. The `cryptography` dependency has had CVE advisories, and users running old versions would not be warned.

**Decision:** Add a `pip-audit` step to the CI workflow after dependency installation. Runs on every push and PR.

### D-9: Duplicate device guard in config flow

**Problem:** The config flow had an `abort.already_configured` message in `strings.json` but no code path that could reach it. A user could add the same iStore device multiple times, creating duplicate entities.

**Decision:** After successful authentication but before `async_create_entry`, check `self._async_current_entries()` for an existing entry with the same `mdm_id`. Abort with `already_configured` if found.

### D-10: Timer time entities enabled by default

**Problem:** Timer schedule time inputs (`time.istore_timer_1_on_time`, etc.) were disabled by default — users had to manually enable them in Settings → Entities. The rationale was that timer switches handled enable/disable.

**Decision:** Re-enable by default. The iStore vendor app surfaces timer schedule configuration, so these entities should be visible. Users who don't use timers can disable them. The timer *binary sensors* (enabled/disabled states) remain disabled as they are redundant with the switches.

### D-11: `_TIMER_POINTS` module-level constant

**Problem:** The timer point list was defined inline in `async_write_timer_settings` (9 strings, 27 lines including formatting). The same list conceptually belongs with `_MEASUREMENT_POINTS` as a declaration of what the integration cares about from the API.

**Decision:** Extract to `_TIMER_POINTS` module-level constant, alongside `_MEASUREMENT_POINTS`. Reduces `async_write_timer_settings` from 55 to 25 lines and makes the timer point set self-documenting.

### D-12: `_coerce` closure standardizes timer value handling

**Problem:** `async_write_timer_settings` had two separate loops with similar but not identical type coercion logic — one for building defaults from coordinator data, one for applying updates.

**Decision:** Extract a `_coerce(point, raw)` closure that handles all three types: time strings (coerced to `str`, default `"00:00"`), integer values (coerced via `int()`, fallback `DEFAULT_WORK_MODE` for WorkMode or `0`), and missing data (same fallbacks). Single source of truth for timer value normalization.

## Consequences

- **Positive:** Config flow auth error classification no longer depends on iStore's exact error wording — typed exceptions are API-change-resilient
- **Positive:** No API response bodies in error messages or logs — defense in depth against potential PII leakage
- **Positive:** One aiohttp session reused across all API calls — fewer TCP+TLS handshakes, less connection churn
- **Positive:** 401 re-auth boilerplate eliminated (6 copies → 1 method)
- **Positive:** Coordinator data access pattern unified across 8 entity classes
- **Positive:** Dependency vulnerability scanning automated in CI
- **Risk:** The `_coerce` closure in `async_write_timer_settings` has no direct unit tests — tested indirectly through entity tests. A dedicated test would improve confidence.
- **Debt:** `number.py` retains ~140 lines of disabled temperature control entity classes as reference documentation. Could be moved to `docs/archived/` or a dead-code branch, but kept in-tree by D-3 of ADR-001.
